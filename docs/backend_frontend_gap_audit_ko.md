# 백엔드-프론트엔드 기능 대조 감사 (재검토판)

## 재검토 목적
이 문서는 이전 감사에서 제기된 3개 항목을 **엄격히 재검증**한 결과다.

재검증 기준을 두 단계로 분리했다.
1. **구현 여부(Strict)**: 프론트엔드에서 어떤 형태로든 호출 가능하면 "구현됨"으로 본다.
2. **제품 완성도(UX Dedicated)**: 일반 사용자를 위한 전용 화면/버튼/플로우가 있는지 본다.

> 핵심: 이전 문서의 "프론트 미구현" 표현은 일부 항목에서 과했다.
> `API 스튜디오`가 OpenAPI를 읽어 GET/POST 엔드포인트를 범용 호출할 수 있으므로,
> 엄격한 의미의 "미구현"이 아니라 "전용 UX 부재"가 더 정확하다.

---

## 공통 근거: API 스튜디오가 전체 GET/POST를 범용 호출함
- `static/NBA.html`에 API 스튜디오 뷰가 존재.
- 설명에 `/openapi.json` 기반 자동 생성 명시.
- `static/app.js`에서 `/openapi.json`을 로드해 GET/POST 엔드포인트를 리스트업하고,
  선택한 엔드포인트를 path/query/body로 실제 호출함.

따라서 아래 3개 항목 모두 "전용 UX 없음"은 맞을 수 있어도,
"프론트에서 아예 구현되지 않음"이라고 단정하려면 API 스튜디오를 제외해야 한다.

---

## 항목별 재판정

### 1) 계약 협상 취소 플로우

- 백엔드: `/api/contracts/negotiation/cancel` 존재.
- 전용 계약 협상 위저드 UI(시작/오퍼/수락/커밋)에는 취소 버튼 없음.
- 그러나 API 스튜디오에서 해당 endpoint를 직접 호출 가능.

**판정**
- Strict 구현 여부: **구현됨(접근 경로 있음)**
- UX Dedicated 여부: **미흡(전용 취소 UX 없음)**

**정정 결론**
- "프론트 미구현" ❌
- "전용 협상 UX 미흡" ✅

---

### 2) 대학/드래프트 탐색 UX

- 백엔드: `meta/teams/team-detail/player/draft-pool` 등 다수 read API 존재.
- 프론트 전용 버튼은 사실상 `/api/college/players` 검색 중심.
- 다만 API 스튜디오에서 대학 관련 read API를 호출해 접근 가능.

**판정**
- Strict 구현 여부: **구현됨(범용 호출 가능)**
- UX Dedicated 여부: **미흡(도메인 전용 탐색 UI 부족)**

**정정 결론**
- "핵심 화면 미구현"(절대적 표현) ❌
- "전용 탐색 UI 부재/부족" ✅

---

### 3) 저장 슬롯 상세 조회 UX

- 백엔드: `/api/game/saves/{slot_id}` 상세 API 존재.
- 런처 저장 슬롯 카드에는 로드/저장만 있고 상세 버튼 없음.
- 하지만 API 스튜디오에서 `/api/game/saves/{slot_id}` 호출 가능.

**판정**
- Strict 구현 여부: **구현됨(범용 호출 가능)**
- UX Dedicated 여부: **미흡(런처 전용 상세 UX 없음)**

**정정 결론**
- "저장 슬롯 상세 미구현" ❌
- "런처 전용 상세 UX 부재" ✅

---

## 최종 요약 (오해 방지용)

- 위 3항목은 모두 **백엔드 미연결/호출 불가 상태는 아님**.
- 다만 상업 출시 관점에서는 API 스튜디오 의존이 크고,
  일반 사용자용 전용 UX가 부족해 **제품 완성도 리스크**가 존재.

즉, 정확한 표현은 다음과 같다.
- "기능 미구현"보다는 **"전용 UX 미흡"**.

## 검증에 사용한 체크
- `rg`로 프론트/백엔드에서 관련 endpoint 및 이벤트 핸들러 존재 여부 확인.
- API 스튜디오 동작 코드(`/openapi.json` 로드 + endpoint 실행 함수) 확인.
- 저장 슬롯/대학/협상 취소 케이스별로 "전용 UI vs 범용 UI" 분리 판정.
