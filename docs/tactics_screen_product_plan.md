# 전술 탭 프론트엔드 고도화 실행 기획안 (상업용 출시 품질)

## 0) 문서 개요
- 목적: 현재 전술 탭 UI를 **상업용 게임 수준의 완성도**로 재설계/재구성한다.
- 대상: `전술 탭` 화면만.
- 구현 범위: **HTML / CSS / JS 한정**.
- 데이터 제약: **기존 조회 API만 사용** (신규 API 추가/응답 스키마 변경 금지).
- 원칙: “개발 툴로 급조한 화면”이 아닌, **브랜드/상태/행동 중심의 제품 UI**로 개선.

---

## 1) 절대 범위 (Scope Lock)

### 포함
1. 전술 탭 레이아웃 재배치
2. 전술 탭 내 컴포넌트 디자인 시스템 적용 (버튼/입력/배지/카드/패널)
3. 전술 탭 상태 표현 개선 (로딩/저장/검증/오류/경고)
4. 전술 탭 내 파생 정보 시각화 (기존 API 데이터 조합 계산)
5. 전술 탭 접근성/키보드 포커스/반응형 개선

### 제외 (절대 수정 금지)
1. 전술 탭 외 다른 탭/페이지의 레이아웃 및 스타일
2. 백엔드 로직/DB 스키마/API 라우트/응답 구조
3. 전역 네비게이션 구조 개편
4. 게임 규칙(시뮬레이션 계산식) 변경

### 안전장치
- CSS는 전술 탭 루트 컨테이너 하위에서만 유효한 네임스페이스 사용 (예: `.tactics-v2 ...`).
- JS는 전술 탭 초기화 함수에서만 동작하도록 진입점 제한.
- 공용 유틸/전역 객체 오염 금지.

## 2) UX 목표 정의

### 제품 목표
- 사용자가 10초 안에 아래 3가지를 파악해야 함:
  1) 현재 전술 방향(공/수 스킴)
  2) 라인업 구성 상태(선발/벤치/분배)
  3) 즉시 조치가 필요한 리스크(경고)

### 체감 목표
- “폼 입력 화면” → “코칭 스태프 전술 콘솔” 전환
- “나열형 데이터” → “결정 지원형 인터페이스” 전환

### 품질 목표 (Done 정의)
- 타이포/간격/라운드/컬러 토큰이 일관적
- 내부 키값(raw enum) 직접 노출 없음
- 마우스/키보드 조작 모두 끊김 없이 가능
- 저장/유효성 상태가 항상 눈에 보임

## 3) 정보 구조 (IA) 및 레이아웃

## 3.1 데스크톱 (기본, 1280px+)
- 상단 고정 헤더 (높이 72~84)
- 본문 3열 그리드
  - 좌측 30%: `스타팅 라인업`
  - 중앙 40%: `로테이션`
  - 우측 30%: `전술 요약/검증`
- 하단 전체폭: `로스터 풀`

## 3.2 태블릿 (768~1279)
- 2열
  - 상단: 스타팅/로테이션 (탭 전환)
  - 하단: 전술 요약 + 로스터

## 3.3 모바일 (<768)
- 1열 스택
- 상단 요약 카드 + 접이식 섹션
- 필수 CTA(`저장`, `시뮬레이션`)는 하단 스티키 바

## 4) 화면 구성 상세

## 4.1 상단 헤더 바
구성 요소:
1. 팀/시즌 아이덴티티
   - 팀명, 시즌 문자열, 보조 라벨
2. 현재 전술 배지
   - 공격 스킴 배지, 수비 스킴 배지
3. 상태 인디케이터
   - `저장됨` / `수정됨` / `검증 오류`
4. 액션
   - 보조: `자동 밸런스`
   - 기본: `저장`
   - 강조: `시뮬레이션 실행`

행동 규칙:
- 값 변경 즉시 상태를 `수정됨`으로 전환.
- 검증 오류 존재 시 `저장` 버튼 비활성 + 이유 툴팁.

## 4.2 스타팅/로테이션 에디터
행(선수 카드) 구성:
- 선수명 + 포지션(보조)
- 공격 역할 셀렉트
- 수비 역할 셀렉트
- 출전 시간 입력(스텝퍼 + 직접 입력)

인터랙션:
- 동일 선수 중복 선택 방지
- 변경된 행은 0.6초 하이라이트
- 역할 변경 시 우측 요약 패널 실시간 갱신

표현 규칙:
- 내부 키(`Engine_Primary`)는 화면에 직접 노출 금지
- 사용자 친화 라벨 맵핑 필수 (예: `1차 볼핸들러`)

## 4.3 전술 요약/검증 패널 (우측)
섹션 A. 분배 검증
- 총합 분(목표 240)
- 스타터 합/벤치 합
- 초과/부족을 색상 및 아이콘으로 표시

섹션 B. 역할 분포
- 공격 역할 카운트
- 수비 역할 카운트
- 과밀/부족 배지

섹션 C. 리스크 알림
- 예: `POA 수비 부족`, `볼핸들러 과다`, `림 압박 부재`
- 룰 기반 경고 (프론트 계산)

섹션 D. 빠른 인사이트
- 1~2줄 추천 문구
- 예: “Spacer가 1명 이하라 하프코트 간격이 좁아질 수 있습니다.”

## 4.4 로스터 풀 패널
- 검색(이름), 필터(포지션)
- 현재 선택된 선수는 선택 불가/비활성 표시
- 클릭 시 해당 슬롯 교체 모드 진입

## 5) 디자인 시스템 스펙 (전술 탭 전용)

## 5.1 컬러 토큰
- 배경: `#0F172A` 계열(다크) 또는 현재 테마 기반 중립톤
- 카드: 배경 대비 +4~6 밝기
- 텍스트:
  - Primary 90~100%
  - Secondary 60~70%
- 상태색:
  - 성공 `#22C55E`
  - 경고 `#F59E0B`
  - 위험 `#EF4444`
  - 정보 `#3B82F6`

## 5.2 타이포 토큰
- H1: 28/700
- H2: 22/700
- Section title: 18/600
- Body: 14~16/400~500
- Caption: 12/400

## 5.3 레이아웃 토큰
- 8pt spacing system 사용
- radius: 10(입력), 12(카드), 16(패널)
- border: 1px 통일
- shadow: 2단계만 허용 (기본/hover)

## 5.4 컴포넌트 규칙
- 버튼: Primary / Secondary / Ghost / Danger
- 입력 높이: 40~44px 통일
- 포커스 링: 2px, 접근성 명확 색상
- 배지: neutral/info/success/warning/danger 5종

## 6) 상태 설계 (State Design)

## 6.1 데이터 상태
1. 초기 로딩
2. 정상 렌더
3. 부분 오류(일부 데이터 누락)
4. 완전 오류(요청 실패)

## 6.2 편집 상태
1. pristine(초기)
2. dirty(수정됨)
3. validating(검증 중)
4. invalid(오류 존재)
5. saved(저장 완료)

## 6.3 피드백 규칙
- 저장 성공: 상단 토스트 + 상태 배지 `저장됨`
- 저장 실패: 토스트 + 실패 사유 + 재시도 버튼
- 분배 오류: 필드 하단 에러 + 우측 패널 경고 동시 노출

## 7) 기존 조회 API 기반 데이터 매핑

> 신규 API 없이, 기존 응답 조합/파생 계산만 사용.

1. 팀/로스터 기본 정보
- 소스: `GET /api/team-detail/{team_id}`
- 사용: 선수명, 포지션, OVR/피로도 등 표시용 메타

2. 리그 날짜/문맥
- 소스: `GET /api/state/summary`
- 사용: 헤더의 날짜/시즌 문맥

3. 다음 경기 문맥
- 소스: `GET /api/team-schedule/{team_id}`
- 사용: 우측 패널 `다음 상대 기준` 보조 텍스트

4. 전술 숙련도/샤프니스
- 소스: `GET /api/readiness/team/{team_id}/familiarity`, `.../sharpness`
- 사용: 인사이트 문구/경고 우선순위 보정

### 프론트 파생 계산 예시
- `minutesTotal = starters.minutes + rotation.minutes`
- `roleCounts[role]++`
- `riskFlags = rules(roleCounts, minutesTotal, positionSpread)`

## 8) 인터랙션 시나리오 (핵심 플로우)

### 플로우 A: 스킴 변경 후 저장
1. 공격 스킴 변경
2. 상태 배지 `수정됨`
3. 우측 인사이트 갱신
4. 저장 클릭
5. 성공 시 `저장됨` + 변경 시각 노출

### 플로우 B: 출전시간 불일치 해결
1. 총합 240 미달/초과 발생
2. 헤더/우측 패널/필드에 동시 경고
3. `자동 밸런스` 클릭 시 보정
4. 경고 해제

### 플로우 C: 선수 교체
1. 로스터 풀에서 선수 선택
2. 대상 슬롯 지정
3. 중복/규칙 검증
4. 성공 반영 + 해당 행 하이라이트

## 9) 구현 가이드 (HTML/CSS/JS)

## 9.1 HTML
- 시맨틱 구조: `header`, `section`, `aside`, `ul/li`, `button`
- 각 패널에 제목/설명/콘텐츠 영역 분리
- 스크린리더용 라벨/설명(`aria-label`, `aria-describedby`) 적용

## 9.2 CSS
- 전술 탭 루트 클래스 네임스페이스 강제
- 토큰(색/간격/폰트)을 CSS 변수로 선언
- 브레이크포인트 3단계(1280/768/480)
- prefers-reduced-motion 대응

## 9.3 JS
- 상태 단일 소스(`tacticsState`) 유지
- 렌더 함수 분리:
  - `renderHeader()`
  - `renderLineupPanels()`
  - `renderSummaryPanel()`
  - `renderRosterPool()`
- 검증 함수 분리:
  - `validateMinutes()`
  - `validateDuplicates()`
  - `computeRiskFlags()`
- 이벤트 위임으로 핸들러 관리 단순화

## 12) 최종 산출물 기준
- 전술 탭은 단순 편집 UI가 아니라, **의사결정 중심 게임 UI**로 체감되어야 한다.
- 사용자는 “무엇이 문제인지/지금 뭘 눌러야 하는지”를 즉시 이해해야 한다.
- 구현 결과는 전술 탭 외 영역에 시각/기능 영향이 없어야 한다.
