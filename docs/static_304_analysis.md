# `/static/NBA.*` 304 로그 분석 및 개선 방향

## 증상 요약
서버 로그에 아래와 같이 `304 Not Modified`가 반복적으로 출력된다.

- `GET /static/NBA.html` → `304`
- `GET /static/NBA.css` → `304`
- `GET /static/NBA.js` → `304`

## 원인 분석
`304 Not Modified`는 **오류가 아니라 정상적인 HTTP 캐시 재검증 응답**이다.
브라우저가 기존에 내려받은 정적 파일(`html/css/js`)의 사본을 가지고 있을 때,
`If-Modified-Since`/`If-None-Match` 등의 조건부 요청을 보내고 서버가 변경 없음으로 판단하면 `304`를 반환한다.

현재 서버는 FastAPI `StaticFiles`로 `/static`을 마운트하고 있어, 정적 파일 제공 시 브라우저 캐시 동작이 일반적인 표준 방식으로 적용된다.

## 왜 자주 보이나?
- 개발 중 페이지 새로고침(F5/Ctrl+R)을 자주 하면 정적 리소스 조건부 요청이 계속 발생한다.
- Uvicorn access log는 모든 요청을 기록하므로, 정상 캐시 트래픽도 "문제"처럼 눈에 띈다.

## 개선 방향

### 1) "오류"로 보지 않도록 팀 가이드 정리 (우선)
- 운영/개발 가이드에 `304`는 정상 캐시 신호임을 명시한다.
- 실제 오류는 `4xx/5xx`, JS 콘솔 에러, API 실패 응답으로 구분한다.

### 2) 개발 편의: 캐시 정책을 환경별 분리
- 개발 환경: `Cache-Control: no-cache` 또는 짧은 `max-age`로 반영 지연 체감 감소
- 운영 환경: 파일명 해시(`NBA.abcd1234.js`) 기반 장기 캐시(`immutable`)로 성능 최적화

### 3) 배포 품질: 캐시 무효화 전략 명확화
- 빌드 시 정적 파일 fingerprinting 도입
- HTML에서 해시된 파일명 참조
- 릴리즈마다 자동 무효화되도록 파이프라인 구성

### 4) 로그 가독성 개선
- 개발 시 access log 레벨/필터 조정(예: `/static` 304 로그 축소)
- 문제 분석 시에는 API 실패 요청만 우선 관찰할 수 있도록 로그 정책 분리

## 빠른 점검 체크리스트
1. 브라우저 개발자도구 Network에서 `Status 304` + `Size (from disk cache/memory cache)` 확인
2. 강력 새로고침(Disable cache) 시 `200`으로 재수신되는지 확인
3. 정적 파일 내용 변경 후 새 요청에서 `200` 또는 최신 내용 반영 확인

## 결론
현재 보이는 `304` 로그 자체는 장애 징후가 아니라, 브라우저 캐시 재검증이 정상 동작하는 신호다.
다만 개발자 경험과 운영 가시성을 위해 **캐시 정책(개발/운영 분리), 무효화 전략, 로그 정책**을 함께 정리하는 것이 좋다.
