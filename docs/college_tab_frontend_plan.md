# 대학 탭 프론트엔드 개선 기획안 (상업용 출시 품질)

## 1) 문서 목적 / 범위

이 문서는 **대학 탭(College League)** 영역만을 대상으로, 현재 HTML/CSS/JS 구조에서 구현 가능한 수준의 고품질 화면 개선 기획안이다.

- 개선 허용 범위: `html`, `css`, `js`
- 개선 대상: 대학 탭 4개 화면
  - 팀 순위
  - 리더보드
  - 빅보드
  - 스카우팅
- 금지 사항: 대학 탭 외 기능/화면 수정 금지
- 데이터 조건: 신규 API 추가 없이 **기존 조회 API**만 사용

---

## 2) 목표 UX / 품질 기준

### 제품 목표
- "게임 UI"가 아닌 "상업용 스포츠 매니지먼트 타이틀" 수준의 밀도와 완성도 달성
- 데이터가 많아도 핵심 정보가 3초 내 파악되도록 정보 계층 재구성
- 바이브 코딩 특유의 랜덤한 스타일을 제거하고, 일관된 디자인 시스템 기반으로 재정렬

### 품질 KPI (UI 측정 기준)
1. 첫인상: 히어로 영역 + 핵심 KPI 가시성 확보
2. 탐색성: 탭 전환/정렬/필터 상태가 항상 명확히 보임
3. 가독성: 테이블의 수치/텍스트 가독성 및 스캔 속도 개선
4. 상태표현: 로딩/빈데이터/오류/선택 상태를 모두 시각화
5. 일관성: spacing, radius, color, typography 규칙 준수

---

## 3) 정보 구조(IA) 재정의

대학 탭은 공통적으로 아래 3레이어로 통일한다.

1. **리그 헤더 레이어**
   - 리그명, 시즌, 대학 수, 예정 드래프트 연도
   - 상단 액션(메인으로)

2. **컨트롤 레이어**
   - 탭(팀 순위/리더보드/빅보드/스카우팅)
   - 화면별 필터/정렬/전문가 선택

3. **콘텐츠 레이어**
   - 기본: 메인 데이터 테이블/리스트
   - 보조: 상세 패널, 요약 카드, 상태 배지

> 핵심 원칙: "모든 화면이 같은 뼈대"를 가지도록 만들어 사용자가 학습한 인터랙션을 재사용 가능하게 한다.

## 4) 비주얼 시스템 가이드 (디자인 토큰 수준)

### 컬러
- `--bg-app`: 앱 배경
- `--bg-surface`: 카드/패널 배경
- `--border-subtle`: 연한 경계
- `--text-primary`: 주요 텍스트
- `--text-secondary`: 보조 텍스트
- `--accent-primary`: 브랜드 강조색
- `--accent-success`: 강점/긍정
- `--accent-warning`: 주의
- `--accent-danger`: 리스크

### 타이포
- H1(페이지): 36/700
- H2(섹션): 28/700
- H3(카드): 22/700
- Body: 15~16/500
- Caption: 12~13/500
- 숫자 컬럼: tabular numerals 적용

### 간격/모서리/그림자
- 8pt spacing 스케일: 4/8/12/16/24/32/40
- radius: 10(기본), 14(큰 카드)
- shadow: 얕은 1단 + hover 2단

### 인터랙션 시간
- hover: 120ms
- panel open/close: 180~220ms
- tab 전환 강조: 150ms

## 5) 공통 컴포넌트 사양

1. **Page Hero**
   - 라벨(영문 소제목) + 한글 타이틀 + 메타 정보(시즌/대학/드래프트)

2. **Segmented Tabs**
   - 현재 선택 탭은 배경 채움 + 하단 인디케이터

3. **KPI Cards (선택형)**
   - 값 + 라벨 + 서브 설명

4. **Data Table Pro**
   - sticky header
   - 정렬 아이콘 상태
   - row hover/selected 상태
   - zebra는 최소화하고 행 간 경계선 정돈

5. **Tag/Badge/Chip**
   - 컨퍼런스, 티어, 상태(진행중/완료/대기), 강점/약점 표현

6. **Empty State**
   - 아이콘 + 1줄 타이틀 + 1줄 설명 + 필요한 경우 CTA

7. **Skeleton Loader**
   - 테이블과 카드에 통일된 스켈레톤 도입

## 6) 화면별 상세 기획

## 6-1) 팀 순위 화면

### 목적
- "리그 판도"를 한 번에 이해하게 만들기

### 레이아웃
- 좌(7): 팀 순위 테이블
- 우(5): 선택 팀 로스터 패널

### 상단 요약(신규 노출, 기존 데이터 기반 계산)
- 1위 팀
- 평균 SRS
- 최고 컨퍼런스 팀(예: East 1위)
- 하위권 컷라인(예: 20위 SRS)

### 테이블 컬럼 정책
- 고정: RK, 팀
- 수치: W/L/SRS 우측 정렬
- 컨퍼런스는 Chip 렌더링
- Top3 행 배지 적용 (Gold/Silver/Bronze 점 표시)

### 로스터 패널
- 선택 팀명 + 컨퍼런스
- 로스터 테이블(선수/포지션/학년/PTS/REB/AST)
- 포지션 분포 요약(텍스트 기반 비율 표기)

---

## 6-2) 대학 리더보드 화면

### 목적
- "누가 현재 대학리그를 지배하는가"를 빠르게 전달

### 상단 컨트롤
- 정렬 기준 버튼군: PTS/REB/AST/STL/BLK
- 현재 정렬 기준 시각 강조

### 테이블 개선
- 랭킹 1~5 배지
- 선수명 우선 강조, 팀명은 보조색
- 수치 컬럼 간 간격 통일
- 동일 팀 반복 시 시각 피로를 줄이기 위해 팀명 타이포 대비 완화

### 보조 정보
- 선택 선수 우측 상세 패널(선택 시)
  - 기본정보(팀/포지션/학년)
  - 핵심 스탯 3개

---

## 6-3) 빅보드 화면

### 목적
- "드래프트 후보 가치"와 "전문가 시각 차이"를 드라마틱하게 전달

### 레이아웃
- 상단: 전문가 선택 세그먼트
- 본문 좌: 랭킹 리스트
- 본문 우: 선수 상세 카드

### 리스트 컬럼
- Rank / 선수 / POS / Tier / 요약(짧게)
- Tier는 색상 배지로 시각화

### 상세 카드(핵심)
- 선수명, 포지션, 티어
- Strengths 태그 2~3개
- Concern 태그 1~2개
- 전문가 코멘트 전문(확장 가능)

### 전문가 비교 (기존 데이터만으로 가능)
- 동일 선수의 전문가별 랭킹 차이(ΔRank) 표시
- 상승/하락 아이콘 표기(색상은 과하지 않게)

---

## 6-4) 스카우팅 화면

### 목적
- "배정 → 진행 → 리포트 확인" 흐름을 명확히

### 레이아웃
- 좌: 스카우터 배정 패널
- 우: 스카우팅 리포트 타임라인/테이블

### 배정 패널
- 스카우터 선택, 대상 선수 선택
- 액션 버튼 우선순위
  - Primary: 배정
  - Secondary: 해제

### 리포트 영역
- 상태 배지(진행중/완료/대기)
- 항목 클릭 시 상세(모달 또는 인라인)
- 빈 상태 강화: 안내문 + 행동 유도 메시지

---

## 7) 상태 설계 (반드시 구현)

1. Loading
   - 최초 조회 시 skeleton
2. Empty
   - 데이터 없음 메시지 + 맥락 설명
3. Error
   - 네트워크/조회 실패 시 재시도 버튼
4. Selected
   - 선택된 행/카드 강조
5. Sorted
   - 현재 정렬 컬럼 표시

---

## 8) 접근성/완성도 체크리스트

- 키보드 포커스 visible
- 대비비율(텍스트/배경) 최소 AA
- hover 외에도 selected/active 상태 명확화
- 숫자/단위 표기 일관성(소수점 자리수 규칙)
- 툴팁/ellipsis 처리로 긴 텍스트 깨짐 방지

---

## 9) 구현 단계(실행 가능한 작업 단위)

### Phase 1 — 기반 구축 (1차)
- 대학 탭 전용 CSS 토큰/레이아웃 정리
- 공통 컴포넌트 클래스 정비(탭/카드/테이블/배지)
- 기존 마크업 구조를 최대한 유지하며 클래스 체계화

### Phase 2 — 화면별 적용 (2차)
- 팀순위: KPI + 좌우 패널 정리
- 리더보드: 정렬 컨트롤 + 랭킹 강조
- 빅보드: 상세 패널 + 태그 구조
- 스카우팅: 상태 배지 + 빈 상태 + 타임라인

### Phase 3 — 인터랙션/품질 마감 (3차)
- 전환 애니메이션, hover, focus polish
- 로딩/에러/빈 상태 정리
- 반응형 레이아웃(최소 1366/1600 기준)

---

## 10) 파일 수정 원칙 (다음 작업자 필독)

1. 대학 탭 관련 HTML/CSS/JS 파일만 수정
2. 공통 파일을 수정해야 할 경우에도 대학 탭 스코프 클래스(`.college-*`)로 영향 범위 제한
3. API 스펙 변경 금지, 기존 조회 API 응답만 사용
4. 비즈니스 로직 변경 금지(표현/정렬/UI 상태 처리만)
5. 다른 탭/다른 페이지 DOM 구조에 영향을 주는 전역 selector 금지

---

## 11) Definition of Done (완료 기준)

- 대학 탭 4개 화면이 동일한 디자인 시스템으로 보인다.
- 각 화면에서 핵심 정보가 시각적으로 우선 노출된다.
- 정렬/선택/상태(로딩/빈/오류) 표현이 모두 구현되어 있다.
- 스카우팅 화면의 빈 상태가 안내형 UX로 개선되어 있다.
- 대학 탭 외 화면에 시각/기능 영향이 없다.

---

## 12) 리스크 및 대응

- 리스크: 기존 CSS가 전역 오염되어 다른 탭에 영향
  - 대응: 대학 탭 루트에 네임스페이스 클래스 부여 후 하위 선택자만 사용
- 리스크: 데이터 필드 편차로 일부 카드가 비어 보임
  - 대응: fallback 텍스트/placeholder 설계
- 리스크: 성능 저하(과도한 DOM 조작)
  - 대응: 테이블 렌더 최소화, 이벤트 위임 사용
