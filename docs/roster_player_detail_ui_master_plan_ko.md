# 팀 로스터 / 선수 상세 화면 UI 마스터 기획안 (상업용 출시 품질 목표)

## 0) 문서 목적 / 적용 범위
- 목적: 현재 팀 로스터 화면과 선수 상세 화면을 **상업용 게임 출시 품질**로 끌어올리기 위한 실행 가능한 UI/UX 제작 사양 정의.
- 범위:
  - 개선 대상: `팀 로스터` 탭, `선수 상세 정보` 탭
  - 작업 가능 파일: `html`, `css`, `js`
  - 금지: 그 외 화면/탭의 구조, 스타일, 동작 변경
- 데이터 원칙:
  - **기존 조회 API만 사용**
  - 신규 저장/수정 API 추가 없음
  - 표시 확장은 기존 필드 재가공/파생 지표 계산으로 해결

---

## 1) 제품 목표와 성공 기준

## 1.1 제품 목표
1. 첫 진입 3초 내에 “팀 상태 + 로스터 핵심 이슈 + 즉시 의사결정 포인트”를 이해시킨다.
2. 선수 상세 화면에서 “이 선수를 어떻게 써야 하는가”를 숫자 나열이 아닌 구조화된 인사이트로 전달한다.
3. 시각 완성도/모션/정렬 일관성을 통해 “바이브 코딩 산출물” 인상을 제거한다.

## 1.2 정량 KPI (권장)
- 로스터 화면 진입 후 선수 상세 클릭률: +25%
- 상세 화면에서 20초 이상 체류 비율: +30%
- 유저 주관 평가(완성도/가독성): 5점 만점 4.3 이상

---

## 2) 현 상태 진단 (문제 정의)

## 2.1 팀 로스터 화면
- 데이터 자체는 충분하나, **표 기반 나열** 중심이라 제품 감성과 정보 계층이 약함.
- 컨디션/경기력 시각 요소가 단순하고 의미 단계(좋음/주의/위험)가 직관적이지 않음.
- 스캔 우선순위(누가 핵심 선수인지, 누가 리스크인지)가 약함.

## 2.2 선수 상세 화면
- 카드 구조는 존재하나, ATTR가 대량 나열되어 **읽기 피로도**가 큼.
- 계약/불만/건강 카드가 분절적으로 보여 “조치 우선순위”가 드러나지 않음.
- 숫자-맥락 연결(강점/약점/권장 역할) 표현이 부족함.

---

## 3) 디자인 콘셉트 / 비주얼 시스템

## 3.1 콘셉트: Front Office Premium (Broadcast + Management)
- 톤: 딥 네이비/차콜 다크 테마 기반
- 강조: 팀 컬러를 CTA/상태 강조/차트 라인에 제한적으로 반영
- 질감: 저강도 글래스 패널 + 정밀한 보더/광원
- 밀도: 정보량은 높게 유지하되, 그룹/우선순위를 강하게 분리

## 3.2 절대 금지 항목
- 기본 브라우저 테이블/버튼 인상 유지
- 화면마다 서로 다른 반경/그림자/애니메이션 사용
- 의미가 없는 강한 색상 남용

---

## 4) 정보 구조 (IA) — 팀 로스터 탭

## 4.1 화면 구조 (상단 → 하단)
1. **Roster Command Bar**
   - 팀명, 시즌, 팀 요약 지표
   - 필터/정렬/검색 컨트롤
2. **Team Snapshot Cards (4개 권장)**
   - 평균 OVR, 평균 나이, 총 샐러리, 리스크 인원 수
3. **Roster Scouting Table**
   - 선수 행 기반 핵심 데이터 + 상태 시각화
4. **Quick Insight Strip**
   - “고연봉 대비 효율 낮음”, “컨디션 저하 Top 3” 등 요약 문장

## 4.2 로스터 표 컬럼 설계 (권장)
- 고정 컬럼: 선수명, 포지션, 나이, 신장, 몸무게, 샐러리, 컨디션, 경기력
- 파생 컬럼(기존 데이터로 계산):
  - 역할 태그(볼핸들러/3&D/빅맨 등 룰 기반)
  - 연봉 효율 인덱스(OVR 대비 연봉 상대지표)
  - 리스크 배지(건강/불만/경기력 저하 신호)

## 4.3 인터랙션
- Hover: 행 배경 밝기 +2~4%, 좌측 팀 컬러 라인 점등
- Active/Selected: 외곽 보더 강조 + 행 고정 하이라이트
- Column Sort: 정렬 방향 아이콘 애니메이션(120~160ms)
- Row Click: 선수 상세 진입 시 shared transition (이름/배지 연계)

## 5) 정보 구조 (IA) — 선수 상세 탭

## 5.1 화면 구조 (상단 → 하단)
1. **Player Hero Panel**
   - 선수명/포지션/신체 + 핵심 KPI + 상태 배지
2. **Primary Insights Row**
   - 역할 요약 문장, 강점 3개, 약점 2개
3. **Attribute Intelligence Panel**
   - ATTR 그룹화 + 그룹별 평균 + 세부 토글
4. **Contract / Morale / Health Action Cards**
   - “현재 조치 필요 여부” 중심 표현
5. **Career Snapshot**
   - 누적/최근 스탯 요약(기존 조회 데이터 범위 내)

## 5.2 Hero Panel 사양
- 좌측: 이름, 포지션 배지, 나이/신체
- 중앙: OVR, 컨디션(%), 경기력(%)
- 우측: 연봉, 차기 시즌 연봉(존재 시), 계약 상태
- 배경: 팀 컬러 기반 저채도 그라데이션 + 번호 워터마크(초상권 안전)

## 5.3 ATTR 그룹화 규칙 (핵심)
- 공격(슈팅/마무리)
- 플레이메이킹(볼핸들/패스/의사결정)
- 수비(온볼/헬프/리바운드/블락)
- 피지컬(스피드/점프/힘/내구성)
- 멘탈(허슬/일관성/적응/야망/로열티 등)

각 그룹에 대해:
- 그룹 점수(평균)
- 등급(Elite / Great / Solid / Risk)
- 상위 2개/하위 2개 하이라이트

## 6) 데이터 표출 정책 (기존 조회 API만 사용)

## 6.1 원칙
- 신규 API 없이 현재 응답값을 조합/가공
- 파생값은 프론트 JS 유틸에서 계산
- 계산식은 화면 전역에서 일관 적용

## 6.2 파생 지표 정의 예시
1. **컨디션 상태 레벨**
   - 85~100: Excellent
   - 70~84: Good
   - 55~69: Caution
   - 0~54: Risk
2. **경기력 상태 레벨**
   - 동일 4단계 체계 사용
3. **연봉 효율 인덱스**
   - `(OVR / Salary) * scale` 형태로 팀 내 백분위 표시
4. **역할 태그**
   - 특정 ATTR 조합 임계치 기반 부여

## 6.3 노출 우선순위
- 1순위: 현재 조치에 직접 영향(건강/불만/컨디션/계약)
- 2순위: 전력 운영 영향(역할/효율/강약점)
- 3순위: 참고 정보(신체/세부 스탯)

## 7) 컴포넌트 설계 (디자이너/프론트 구현 기준)

## 7.1 공통 컴포넌트
- `StatusPill` (Good/Caution/Risk)

## 7.2 로스터 전용 컴포넌트

- `RosterTableHeader` (고정/정렬)
- `RosterRow`
- `RosterQuickInsight`

## 7.3 상세 전용 컴포넌트
- `PlayerHero`
- `InsightSummary`
- `AttrGroupPanel`
- `ActionStatusCard`
- `CareerMiniTable`

## 8) 디자인 토큰 (초안)

## 8.1 Color
- `bg.base = #0A1120`
- `bg.elevated = #121A2B`
- `bg.card = #182236`
- `text.primary = #F1F5FF`
- `text.secondary = #95A3BD`
- `border.soft = #2A3A55`
- `state.good = #22C55E`
- `state.caution = #F59E0B`
- `state.risk = #EF4444`

## 8.2 Typography
- Page Title: 42/50, 800
- Section Title: 24/32, 700
- Body: 15/22, 500
- Caption: 12/16, 500
- Numeric: tabular-nums + 600

## 8.3 Radius / Shadow / Spacing
- Radius: card 16 / control 10 / pill 999
- Shadow:
  - card: `0 8 24 rgba(0,0,0,0.28)`
  - hover: `0 12 32 rgba(0,0,0,0.34)`
- Spacing Scale: 4, 8, 12, 16, 24, 32, 40

---

## 9) 모션/인터랙션 규격
- Hover: 160~200ms ease-out
- Panel Enter: 220~280ms cubic-bezier(0.2, 0.8, 0.2, 1)
- Tab/Filter Change: 140~180ms
- 원칙:
  - 모든 모션은 정보 이해 보조 목적
  - 과한 패럴랙스/글로우 금지

---

## 10) 접근성 / 국제화 / 숫자 포맷
- 키보드 포커스: 2px 고대비 포커스 링
- 색맹 대응: 색상 + 아이콘/텍스트 동시 표기
- 숫자:
  - 샐러리: 천단위 구분 + `$`
  - 퍼센트: 정수 반올림 표기
  - 신장/체중: 포맷 일관성 유지
- 다국어 확장 고려:
  - 라벨 최소 너비 확보
  - 긴 텍스트 ellipsis + tooltip

---

## 11) 상태 설계 (반드시 구현)

각 인터랙티브 요소별 상태 정의:
- default
- hover
- active(pressed)
- focus-visible
- disabled
- loading(필요 시)

## 11.1 Skeleton
- 로스터: 헤더 + 8행 skeleton
- 상세: Hero + 그룹 카드 3개 skeleton

## 11.2 Empty
- 로스터 비어있음: 원인 설명 + 재시도 CTA
- 스탯 없음: “아직 누적 데이터가 없습니다” 안내

## 11.3 Error
- API 실패: 섹션 단위 에러 카드 + 재시도 버튼
- 일부 데이터 누락: 사용 가능한 필드만 렌더(깨짐 금지)

## 12) 구현 가드레일 (이번 작업 제약)
1. **수정 범위 제한 준수**
   - 팀 로스터/선수 상세 HTML/CSS/JS만 수정
   - 다른 탭, 공용 라우팅, 백엔드 로직 불가
2. **API 제약 준수**
   - 기존 조회 API 외 신규 엔드포인트/파라미터 의존 금지
3. **성능 가드레일**
   - 초기 렌더 16ms 프레임 예산 고려
   - 대량 DOM 업데이트는 배치 처리
4. **기술 부채 방지**
   - 인라인 스타일 최소화
   - 토큰/유틸 함수로 중복 제거

## 13) 프론트 구현 단계 로드맵 (실행 순서)

## Phase 1 — Foundation
- 디자인 토큰/타이포/간격 시스템 도입
- 공통 컴포넌트 뼈대 구축

## Phase 2 — Roster UI
- Command Bar + Snapshot 카드 + Table 리뉴얼
- 필터/정렬/선택 상태/행 인터랙션 고도화

## Phase 3 — Player Detail UI
- Hero 재구성 + ATTR 그룹화 + 액션 카드 재정렬
- 자동 요약 문장/강약점 생성 적용

## Phase 4 — Polish
- Skeleton/Empty/Error 정교화
- 모션/미세 인터랙션/접근성 점검
- 반응형(최소 1366 기준 + 하위 폭 대응)
