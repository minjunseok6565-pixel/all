# NBA 시뮬레이션 게임 순위표 탭 UI 고도화 기획안 (상업 출시 품질)

## 0) 문서 목적 / 적용 범위
- 목적: 현재 `순위표` 화면을 상업용 게임 수준의 완성도로 끌어올리기 위한 **실행 가능한 프론트엔드 기획안**을 제공한다.
- 적용 범위: `순위표 탭` 한정.
- 수정 가능 파일: `static/NBA.html`, `static/NBA.css`, `static/NBA.js`.
- 금지 사항: 순위표 탭 외 다른 화면/기능 변경 금지.
- 데이터 제약: **기존 조회 API만 사용**(신규 API/스키마 추가 금지).

---

## 1) 제품 목표 정의

### 1-1. 경험 목표
1. 첫인상 2초 내 "리그 운영 대시보드" 느낌 전달.
2. 5초 내 핵심 정보(순위 구간, 격차, 최근 폼) 파악 가능.
3. 표가 아니라 "경쟁 구도"가 보이도록 시각 계층화.

### 1-2. 품질 목표 (상업용 기준)
- 일관된 디자인 시스템(타이포, 간격, 색상, 상태) 적용.
- 정보 밀도는 유지하되 가독성 저하 없이 스캔 속도 향상.
- 마이크로 인터랙션은 절제되게 적용(과한 효과 금지).
- 접근성 최소 기준 준수(명도 대비, 키보드 포커스, 상태 전달).

---

## 2) UX 원칙 (이번 탭의 설계 철학)
1. **Context First**: 사용자가 어디를 보고 있는지(시즌/컨퍼런스/정렬 기준) 먼저 보이게.
2. **Tier Visibility**: 플레이오프/플레이인/탈락권을 즉시 식별 가능하게.
3. **Progressive Disclosure**: 핵심 지표 우선 노출, 보조 지표는 확장 제공.
4. **Numeric Legibility**: 숫자 중심 화면에 최적화된 타이포/정렬 규칙 강제.
5. **No New Data Dependency**: 기존 API 필드 조합/계산으로만 정보 확장.

## 3) 화면 정보 구조(IA)

### 3-1. 상단 Hero 영역
- 좌측
  - 타이틀: `STANDINGS`
  - 서브타이틀: `Eastern Conference · 2025-26 Regular Season`
- 중앙 요약 칩 (기존 데이터 기반 계산)
  - `Top Offense`: PPG 최대 팀
  - `Top Defense`: OPP PPG 최소 팀
  - `Best Diff`: DIFF 최대 팀
- 우측 컨트롤
  - 컨퍼런스 토글 (East/West)
  - 정렬 기준 드롭다운 (PCT 기본, DIFF, STRK, L10)
  - `메인으로` 버튼(보조 버튼 스타일)

### 3-2. 순위표 본문 영역
- 고정 상단 헤더 + 스크롤 가능한 본문
- 행 좌측 티어 바
  - 1~6위: Playoff Direct
  - 7~10위: Play-In
  - 11~15위: Out
- 기본 컬럼(항상 표시)
  - RK, TEAM, W-L, PCT, GB, STRK, L10, DIFF
- 확장 컬럼(접기/펼치기)
  - HOME, AWAY, DIV, CONF, PPG, OPP PPG

### 3-3. 보조 정보
- 하단 legend
  - 색상별 티어 의미 설명
- 데이터 기준 시점 표기
  - `정규시즌 확정 경기 기준`

## 4) 데이터 매핑 (기존 조회 API 필드만 사용)

> 아래는 “새 API 없이” 가능한 파생 처리 규칙.

1. `W-L`: `W` + `L` 문자열 결합.
2. `Games Played`: `W + L`.
3. `Top Offense`: 최대 `PPG` 팀 선택.
4. `Top Defense`: 최소 `OPP PPG` 팀 선택.
5. `Best Diff`: 최대 `DIFF` 팀 선택.
6. `Playoff/Play-In/Out` 구간: 현재 `RK` 기준 분류.
7. `STRK` 배지 상태: `W*`/`L*` 접두로 색상 분기.

## 5) 컴포넌트 상세 기획

### 5-1. Hero Header
- 높이: 120~144px
- 구성: 타이틀 블록 + 칩 3개 + 컨트롤 블록
- 배경: 약한 그라데이션 + 얕은 보더
- 목적: 테이블 시작 전 문맥/상태 제공

### 5-2. Summary Chip
- 내용: 지표명 + 팀명 + 값
- 예시: `Top Offense | BOS | 121.4`
- 상태: hover 시 배경 명도 +4% 변화

### 5-3. Conference Toggle
- 세그먼트 컨트롤 형태
- 선택 상태 대비 명확화(채움 + 텍스트 색 반전)
- 전환 시 테이블 150~200ms fade

### 5-4. Sort Dropdown
- 기본값 `PCT`
- 정렬 방향 토글 포함 (↑/↓)
- 정렬 적용 시 헤더에 아이콘 유지

### 5-5. Standings Table
- 헤더 고정(sticky)
- 행 높이 52~56px
- zebra 배경(아주 약한 대비)
- hover 상태: 행 배경 + 좌측 티어바 강조

### 5-6. Tier Bar
- 각 행 좌측 4px 바
- 색상 토큰
  - Playoff: `--tier-playoff`
  - Play-In: `--tier-playin`
  - Out: `--tier-out`

### 5-7. Team Cell
- 팀 배지(이니셜 원형) + 팀명 + 선택 아이콘(옵션)
- 긴 팀명 말줄임 + tooltip

### 5-8. Stat Badge
- STRK, DIFF 등 상태형 숫자를 배지로 표현
- `Wn`: 긍정 배지, `Ln`: 부정 배지
- `DIFF > 0`: + 기호 표시 및 긍정 색상

## 6) 디자인 시스템 제안

### 6-1. 색상 토큰
- `--bg-page`: 페이지 배경
- `--bg-surface`: 카드/테이블 배경
- `--text-primary`, `--text-secondary`
- `--accent`: 리그 브랜드 포인트
- `--positive`, `--negative`, `--neutral`
- `--tier-playoff`, `--tier-playin`, `--tier-out`

### 6-2. 타이포 토큰
- Display Title: 32/700
- Section Title: 24/700
- Table Header: 13/600 + letter spacing
- Body Stat: 22~24 폭 기준 숫자 가독
- Caption: 12~13/500
- 숫자 계열: `font-variant-numeric: tabular-nums`

### 6-3. 간격/반경/보더
- 8pt spacing scale: 8, 12, 16, 24, 32
- radius: 10~14
- border: 1px low-contrast
- shadow: low elevation 1단계만 사용

---

## 7) 인터랙션/모션 가이드

### 7-1. 모션 원칙
- 짧고 기능적인 모션(150~200ms)
- 장식용 애니메이션 금지

### 7-2. 이벤트별 모션
1. 컨퍼런스 전환: 테이블 fade-out/in.
2. 정렬 변경: 헤더 아이콘 전환 + 본문 위치 재배치.
3. row hover: 배경 톤 + 좌측 티어바 두께 미세 증가.
4. 로딩 상태: skeleton row 6~8개 표시.

---

## 8) 반응형 전략

### 8-1. 1440px 이상
- 전체 컬럼 노출(확장 컬럼 기본 펼침 가능)

### 8-2. 1024~1439px
- 확장 컬럼 기본 접힘
- 핵심 컬럼 고정

### 8-3. 768~1023px
- 가로 스크롤 허용 + 첫 2열 sticky (RK, TEAM)

### 8-4. 767px 이하
- 카드형 리스트로 변환하지 말고, 테이블 유지 + 수평 스크롤
- 이유: 순위 비교 정확도 유지

---

## 9) 접근성/로컬라이제이션
1. 텍스트 대비 WCAG AA 수준 목표.
2. 키보드 포커스 링 명시.
3. 색상 외 수단으로 상태 전달(아이콘/레이블 병행).
4. 한글/영문 혼합 대비 line-height 여유 확보.
5. 숫자 단위/소수점 표기 일관성 유지.

## 11) 품질 검수 기준 (Definition of Done)

### 11-1. 시각 품질
- 첫 화면에서 컨퍼런스/시즌/핵심 랭킹 맥락 인지 가능
- 테이블이 "정보 계층"을 가지며 평면적이지 않음
- 브랜딩/폰트/간격의 일관성이 깨지지 않음

### 11-2. 기능 품질
- 기존 API만으로 모든 UI 정보 생성 가능
- 정렬/토글/확장 컬럼 동작이 안정적
- 로딩/빈 상태/오류 상태 시 화면 붕괴 없음

### 11-3. 성능/유지보수
- 불필요한 DOM 재생성 최소화
- 스타일 토큰 기반으로 후속 화면 확장 가능
- JS 로직이 데이터-뷰 분리 구조를 갖춤

## 12) 단계별 실행 플랜

### Phase 1 (1차 완성도 확보)
- 테이블 구조 재편(기본/확장 컬럼)
- 티어바/배지/숫자 가독성 규칙 적용
- sticky header와 hover 상태 구현

### Phase 2 (상업용 감성 강화)
- Hero + summary chip + 토글/정렬 UI
- 컬러/타이포/간격 토큰 정비
- 로딩 skeleton 도입

### Phase 3 (폴리싱)
- 모션/전환 미세 조정
- 접근성 검수 및 대비 보정
- 반응형 예외 케이스 정리

## 13) 리스크 및 대응
1. **컬럼 과밀 리스크**: 확장 컬럼 접힘 기본값으로 완화.
2. **모션 과다 리스크**: 지속시간 상한(200ms) 강제.
3. **데이터 결측 리스크**: null/0/- 포맷 규칙 선 정의.
4. **다른 탭 영향 리스크**: CSS/JS 네임스페이스를 standings scope로 한정.

## 14) 다음 작업자 인수인계 메모
- 이번 문서는 "실제 구현 직전" 수준의 기준안이다.

- "멋있어 보이는 것"보다 "빠르게 읽히는 것"을 우선할 것.
